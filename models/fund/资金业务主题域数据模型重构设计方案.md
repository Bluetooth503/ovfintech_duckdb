# 资金业务主题域数据模型重构设计方案（更新版）

> 基于生产环境 `fund_workflow` 工作流（72 SQL 任务 / 25 目标表）的专项重构设计
> 设计目标：根治命名混乱、重复计算、跳层、指标口径不一致、DWS 薄弱等系统性债务
> **更新日期：2026-04-22**
> **当前状态：核心状态表、快照表、日聚合表已实现并验证通过**

---

## 一、现有资金域问题诊断

### 1.1 命名规范极度混乱

| 现有表名 | 实际含义 | 命名问题 |
|---------|---------|---------|
| `dwd_fund_billing_i_d` | 账单明细（增量） | `_i_d` 混合粒度和策略，应为 `_fact_i` |
| `dwd_fund_risk_report_f_d` | 风险报告明细 | DWD 层用 `_f_d`（全量日），与增量语义矛盾 |
| `dws_fund_credit` | 授信宽表 | **无后缀**，无法判断更新策略和粒度 |
| `dws_fund_debt_daily` | 借据日汇总 | `_daily` 与项目规范的 `_df`/`_di` 混用 |
| `dws_fund_member_daily` | 客户日汇总 | 同上，且被 6 个任务重复写入 |
| `ads_rpt_fund_member_1d_d` | 客户资金日报 | `_1d_d` 与 `_daily` 同时存在，含义重叠 |
| `ads_rpt_fund_member_1m_m` | 客户资金月报 | `_1m_m` 冗余（月度+月），应为 `_mf` |

**结论：同一工作流内存在 5 种以上命名风格，新人无法从表名读取粒度、策略、业务含义。**

### 1.2 重复计算泛滥

| 目标表 | 写入任务数 | 根因 |
|--------|-----------|------|
| `ads_rpt_fund_member_1d_d` | **8 个** | 8 个任务 SQL 相似度 >95%，仅数据源分区不同 |
| `dws_fund_member_daily` | **6 个** | 历史数据重跑与日常增量未分离，同逻辑跑多遍 |
| `dws_fund_repay_daily` | **5 个** | "今日还款"与"还款数据重新加载"逻辑几乎相同 |
| `dws_fund_loan_daily` | **5 个** | 同上 |

**结论：历史回溯与日常增量混在一个工作流，无单一事实来源，数据覆盖风险极高。**

### 1.3 DWS 层职责不清，ADS 层负担过重

现有 DWS 层的问题：

- `dws_fund_credit`：**0 个 JOIN**，只是 DWD 的简单投影 + `credit_loan_balance = credit_quota - remain_quota`。这不是"汇总"，是"透传"。
- `dws_fund_loan_daily` / `dws_fund_repay_daily`：只做账单聚合（`GROUP BY bill_date, loan_no`），然后关联借据→授信获取客户信息。复杂度未收敛，ADS 仍需反复关联。
- `dws_fund_member_daily`：聚合客户级放款/还款/授信/余额/累计值，但**累计指标（year_cumulative_lending 等）在此层用 COALESCE 拼凑，逻辑散落在多个任务中**。

ADS 层的问题：

- `ads_rpt_fund_member_1d_d`：组合 `dws_fund_member_daily` + `dws_fund_member_offline_daily` + 昨日余额关联 + 年/总累计值计算。**一个 ADS 报表承担了 DWS 层的汇总职责**。
- `ads_rpt_fund_zd_registe_f_d`：**跳层**直接查 `dwd_fund_zd_registe_f_d`。

### 1.4 指标口径不一致

生产环境中 `loan_balance` 在资金域至少存在 **3 种口径**：

| 表名 | `loan_balance` 计算逻辑 | 实际含义 |
|------|----------------------|---------|
| `dws_fund_credit` | `credit_quota - remain_quota` | 授信维度已用额度 |
| `dwd_fund_debt_i_d` | 直接取 `loan_balance` 字段 | 借据维度未还本金 |
| `dws_fund_member_credit_daily` | `SUM(loan_balance)`（按授信） | 授信维度汇总借据余额 |

**结论：同一指标名在不同层级代表不同业务概念，导致 ADS 层组合时数值对不上。**

### 1.5 缺少统一维度设计（资金域未复用已有维度）

- **资金域 DIM 层表数量为 0**，但 `models/customer/dim/` 已存在完善的客户维度表：`dim_customer_unify`（客户统一视图）、`dim_customer_scene_rel`（含资金方信息）、`dim_scene`（场景层级）
- 生产环境中资金域每个 DWS/ADS 自行 LEFT JOIN 客户信息，未复用已有维度
- 客户信息变更（如企业改名）需要修改所有引用该维度的任务
- 资金域特有的金融产品、银行维度缺失

## 二、重构设计原则

### 2.1 命名规范统一

严格遵循项目规范：`{layer}_{domain}_{subject}[_{process}]_{table_type}_{update_mode}.sql`

| 层级 | 表类型 | 更新模式 | 示例 |
|------|--------|---------|------|
| `dim` | — | 无后缀（全量） | `dim_fund_financial_product.sql` |
| `dwd` | `fact` | `_i` 事件级增量 | `dwd_fund_credit_fact_i.sql` |
| `dws` | `state` | `_state_df` 日全量（当前状态，不保留历史） | `dws_fund_credit_state_df.sql` ✅ |
| `dws` | `snap` | `_snap_df` 日快照（保留历史） | `dws_fund_loan_balance_snap_df.sql` ✅ |
| `dws` | `agg` | `_di` 日增量 / `_mi` 月增量 / `_wi` 周增量 / `_df` 日全量 | `dws_fund_customer_agg_df.sql` ✅ |
| `ads` | `dashboard` | `_mf` 月全量 | `ads_fund_operation_dashboard_mf.sql` |

### 2.2 分层边界严格

```
ODS ──→ DWD ──→ DWS ──→ ADS
           ↑      ↑
           └──── DIM（可被 DWD/DWS/ADS 引用）
```

- **DWD**：只做清洗、标准化、去重、关联维度获取代理键。**不做聚合**。
- **DWS**：承担所有汇总、累计、比率计算。**同一指标在 DWS 层只定义一次**。
- **ADS**：只做**轻量组装**（SELECT + JOIN + 少量过滤），单 SQL JOIN ≤ 3。
- **禁止跳层**：ADS 不直接引用 DWD，DWD 不反向依赖 DWS。

### 2.3 消除重复计算

- **每张表只有一个 dbt 模型产出**
- 历史回溯通过 dbt `--vars` 或 `full-refresh` 实现，不复制任务
- 累计指标（如 year_cumulative_lending）在 DWS 层用窗口函数自关联计算，不在 ADS 层拼凑

### 2.4 指标口径统一管理

- 在 DWS 层建立**指标定义清单**，明确每个指标的业务含义、计算逻辑、来源表
- `loan_balance` 统一拆分为：
  - `credit_used_quota` = 授信已用额度（授信维度）
  - `loan_balance` = 借据未还本金（借据维度）
  - `total_loan_balance` = 客户总贷款余额（客户维度汇总借据）

### 2.5 维度统一设计（复用 customer 域，补充资金域特有维度）

- **复用 customer 域维度**：`dim_customer_unify`（客户统一视图）、`dim_customer_scene_rel`（含 fund_org_id / fund_org_name）、`dim_scene`（场景层级）
- **资金域仅补充特有维度**：金融产品维度（`dim_fund_financial_product`）、银行维度（`dim_fund_bank`）
- 变更一处全局生效，禁止各模型自行 JOIN ODS 取客户信息

### 2.6 状态表与快照表的区别

**状态表（`_state_df`）**：
- 用途：记录实体的当前最新状态（如 T-1 日的余额）
- 更新方式：日全量覆盖，不保留历史数据
- 粒度：实体ID（如 customer_id、credit_id）
- 适用场景：只需最新状态，不需要历史趋势分析

**快照表（`_snap_df`）**：
- 用途：记录实体的历史状态变化（如每日余额历史）
- 更新方式：按日期追加，保留完整历史数据
- 粒度：实体ID + 统计日期（如 customer_id + stats_date）
- 适用场景：需要历史趋势分析、计算环比、还原任意时点状态

**选择原则**：
- 需要历史趋势 → 使用快照表
- 只需当前状态 → 使用状态表
- 快照表数据量大，状态表数据量小

## 四、已实现模型详细说明

### 4.1 状态表实现要点

#### dws_fund_credit_state_df ✅

**实现要点**：
- 使用 `TRY_STRPTIME(credit_end_date, '%Y%m%d')` 处理日期格式
- 计算用信率：`credit_used_quota / credit_quota * 100%`
- 标识有效授信：`credit_result = '1' AND credit_end_date >= stats_date`
- T-1日状态：只统计 `trx_date <= T-1` 的数据

**输出字段**：
- 授信基础信息：credit_id, customer_id, credit_quota, remain_quota, credit_used_quota
- 利率信息：base_rate, rate_float
- 期限信息：credit_start_date, credit_end_date
- 计算字段：utilization_rate, is_valid_credit, has_balance

#### dws_fund_promissory_note_state_df ✅

**实现要点**：
- 关联授信表获取客户信息：通过 contract_code 关联
- 到期天数计算：`DATEDIFF('day', stats_date, promissory_note_end_date)`
- 逾期判断：promissory_note_end_date < stats_date
- 到期预警：7天内到期、30天内到期标识

**输出字段**：
- 借据基础信息：promissory_note_no, loan_balance, credit_quota, interest_rate
- 客户信息：customer_id, customer_name（来自授信表）
- 到期信息：days_to_maturity, is_overdue, is_due_within_7d, is_due_within_30d

#### dws_fund_customer_loan_balance_state_df ✅

**实现要点**：
- 整合授信和借据两个数据源
- 贷款余额统计：只统计 loan_balance > 0 的借据
- 客户维度汇总：按客户ID汇总所有授信额度和贷款余额

**输出字段**：
- 客户授信：total_credit_quota, total_remain_quota
- 客户贷款：total_loan_balance, is_loan_customer

### 4.2 快照表实现要点

- 到期预警：due_7d_loan_balance, due_7d_promissory_note_cnt, due_30d_loan_balance, due_30d_promissory_note_cnt
- 比率指标：overdue_rate_pct

## 五、技术实现亮点

### 5.1 字段命名统一

**解决指标口径不一致问题**：

| 原字段名 | 统一后字段名 | 说明 |
|---------|-------------|------|
| `loan_balance`（授信表） | `credit_used_quota` | 授信已用额度（授信维度） |
| `loan_balance`（借据表） | `loan_balance` | 借据未还本金（借据维度） |
| `credit_loan_balance` | `total_loan_balance` | 客户总贷款余额（客户维度） |

### 5.2 日期处理优化

**统一处理多种日期格式**：
- YYYYMMDD格式：使用 `TRY_STRPTIME(date_str, '%Y%m%d')`
- TIMESTAMP类型：使用 `CAST(timestamp_field AS DATE)`
- DATE类型：直接比较

### 5.3 关联逻辑优化

**解决客户信息获取问题**：
- 借据表通过 contract_code 关联授信表获取客户信息
- 使用 LEFT JOIN 确保数据完整性
- 使用 ROW_NUMBER() 窗口函数取最新状态

### 5.4 累计指标实现

**在DWS层使用窗口函数**：
```sql
SUM(daily_loan_amt) OVER (PARTITION BY customer_id ORDER BY stats_date
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS cumulative_loan_amt
```

### 5.5 历史数据保留

**状态表 vs 快照表**：
- 状态表：日全量覆盖，不保留历史（_state_df）
- 快照表：按日期追加，保留完整历史（_snap_df）
